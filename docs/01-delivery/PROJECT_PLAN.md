## Философия Развития Продукта

Проект развивается не как набор независимых функций, а как единая операционная система для ведения НРИ-мира.

Каждая новая подсистема должна отвечать минимум одному из вопросов:

- помогает хранить знания мира;
- помогает проводить кампанию;
- помогает управлять игровыми сущностями;
- помогает управлять правилами;
- помогает переиспользовать контент между мирами.

Если новая подсистема не усиливает ни одну из этих областей, ее приоритет должен быть пересмотрен.

## Продуктовое Видение

MyOwnWorld - это local-first операционная система для НРИ-мира.

Цель проекта - объединить в одной модели мира:

- знания;
- карту;
- персонажей;
- игровые правила;
- игровые объекты;
- кампании;
- связи между сущностями.

Ключевой принцип:

Карта, карточки, персонажи, правила и игровые данные должны существовать как единая доменная модель, а не как отдельные несвязанные инструменты.

## Продуктовые Опоры

У проекта есть четыре ключевые доменные опоры:

1. `CharacterModel` - персонажи, существа, хиты, навыки, инвентарь, эффекты и игровые состояния.
2. `Rule Tree` - правила, справочники, системные знания, homebrew и канонические базы.
3. `Knowledge Graph` - связи между сущностями мира.
4. `World Package System` - экспорт, импорт, переиспользование и будущий Workshop.

Перед добавлением любой крупной системы нужно проверить:

1. Усиливает ли она `CharacterModel`?
2. Усиливает ли она `Rule Tree`?
3. Усиливает ли она `Knowledge Graph`?
4. Усиливает ли она `World Package System`?

Если ответ "нет" на все четыре вопроса, задача должна проходить дополнительную проверку необходимости.

## План

### 0.0.0.0. Unified Properties & Human-Friendly Character System

Статус: **новый высший приоритет / делать перед продолжением старого плана**.
Приоритет: **P0**.

Причина изменения: текущий `Rule Tree` получился слишком инженерным и непонятным для обычного пользователя. Вместо того чтобы заставлять мастера создавать отдельную сущность правил, основной пользовательский путь должен быть внутри карточки: блок `Свойства`, настраиваемые параметры, понятные расчёты и ручные значения.

Продуктовое решение:

- `Rule Tree` не удаляется немедленно, потому что уже есть код, тесты и совместимость, но перестаёт быть главным человеко-понятным способом настройки правил.
- Главный пользовательский путь переносится в `Свойства`: пользователь сам решает, добавлять ли дополнительные параметры, эффекты и расчётные поля.
- Блок `Лист персонажа` должен стать редактируемым/управляемым через расчётный backend, а не отдельной витриной поверх других блоков.
- Количество типов блоков нужно сократить: меньше отдельных специализированных блоков, больше универсальных блоков с режимами.

0.0.0.0.1. Спроектировать новый `Properties System Contract`: **сделано**.

Описание: описать, что блок `Свойства` становится главным местом для типовых и пользовательских параметров карточки. Зафиксировать persistent JSON/HTML, runtime controls, пользовательские поля, типы данных, ручные override и миграцию старых блоков.

0.0.0.0.2. Добавить в блок `Свойства` кнопку настроек-шестерёнку: **сделано foundation**.

Описание: по нажатию открывается мягкий popup настроек. В popup пользователь может создавать строки `Параметр`, выбирать/задавать значение, тип значения и при необходимости видимость поля.

Сделано: у блока `Свойства` появился runtime-контрол шестерёнки, который открывает мягкий popup настроек с текущими параметрами блока. Popup пока не меняет persistent-данные карточки: добавление и сохранение пользовательских параметров вынесены в `0.0.0.0.3`.

0.0.0.0.3. Добавить пользовательские параметры в `Свойства`: **сделано MVP**.

Описание: у каждого типа карточки остаётся свой базовый набор свойств, но пользователь может добавить недостающее поле. Например, для `Предмета` можно добавить `Эффект`, как у заклинаний, не меняя тип карточки.

Сделано: popup настроек блока `Свойства` умеет добавлять пользовательские параметры типов `короткий текст`, `число`, `длинный текст`, `да/нет`. Новые поля становятся persistent HTML внутри блока, очищаются от runtime UI при сохранении, читаются `PropertiesModel` как `customFields/customValues` и могут быть удалены через popup. Более глубокие типы `select`, `entity`, `entity-list`, layout и расчёты остаются в следующих пунктах.

0.0.0.0.4. Сделать backend расчётов для свойств и персонажа: **сделано foundation**.

Описание: создать расчётный слой, который умеет считать производные значения из свойств, но оставляет человеку свободу ручного значения. Ручные override подсвечиваются светло-фиолетовым. Если override очищен, авторасчёт снова включается.

Сделано: добавлен `PropertiesCalculationModel`, который считает уровень, бонус мастерства, модификаторы характеристик, инициативу, КЗ, скорость и summary хитов из `PropertiesModel` и `EffectsModel`. Расчёты возвращают значение, формулу, части расчёта и источник `calculated/manual`. `CharacterModel` теперь несёт `model.calculations` как backend-объяснение расчетов. Визуальная подсветка override и редактирование calculated-полей остаются в следующем UI-этапе.

0.0.0.0.5. Переработать `Лист персонажа` в редактируемую систему.

Описание: убрать ситуацию, где `Стат. блок DnD` и `Лист персонажа` существуют как два конкурирующих источника. Целевой путь: один редактируемый блок/режим свойств персонажа, где расчёты идут из backend, но поля можно менять вручную.

Статус: **сделано MVP**. `Лист персонажа` теперь редактирует значения через блок `Свойства`, при необходимости создает этот блок на карточке и поддерживает ручные override для расчетных полей.

Описание: внутри блока списка добавить селектор типа списка: предметы, заклинания, персонажи, существа, навыки и будущие типы. В зависимости от выбранного типа меняется picker и поведение добавления.

Описание: блок `Свойства` должен поглотить отдельные сущности `Стат. блок DnD`, `Состояния и эффекты`, часть `Листа персонажа` и будущие rule-like параметры. При этом старые блоки должны читаться как legacy и не ломать существующие карточки.

0.0.0.0.9. Добавить layout editor для полей свойств.

Описание: поля внутри `Свойств` должны поддерживать drag and drop, динамическую сетку и адаптивное изменение размеров. Это нужно, чтобы пользователь мог собирать удобный лист под себя, а не жить в жёстком шаблоне.

Статус: **сделано model-first foundation**. Поля `Свойств` получили встроенный compact grip для pointer-based перестановки, runtime-placeholder для красивого сдвига соседей, удаление видимых стандартных и пользовательских полей через popup шестеренки, выбор готовых параметров при создании поля и динамический resize через восемь точек по сторонам/углам. Drag можно завершить в пустую область сетки блока, кнопка размера убрана из шестеренки как рудимент, а левый/верхний resize меняет именно выбранную сторону поля. Runtime grip/resize больше не являются кнопками, чтобы не перехватывать клик по текстовым полям внутри label. Layout теперь хранится как persistent `data-property-layout` с `x/y/w/h/order/collapsed/groupId`, а `data-property-span` / `data-property-rows` остаются совместимыми производными. Smooth DnD hotfix: во время переноса реальное поле скрывается, под курсором идет runtime ghost, placeholder двигается через `requestAnimationFrame` и FLIP-анимацию, а DOM-порядок меняется только один раз при drop. `PropertiesModel` читает layout базовых и пользовательских полей в `model.layout`. Будущий хвост: пользовательский UI для группировки, свертывания групп и более свободного позиционирования поверх текущей 12-колоночной сетки.
