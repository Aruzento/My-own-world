# Design System Contract

Дата обновления: 05.06.2026

## Назначение

Дизайн-система MyOwnWorld нужна, чтобы интерфейс развивался цельно, а не как набор случайных CSS-правок. Продукт должен ощущаться как рабочая среда мастера НРИ: темная, спокойная, современная, удобная для долгой работы и без лишней декоративности.

Этот контракт не требует полного редизайна сразу. Он задает правила, по которым новые UI-элементы и будущие этапы модернизации должны добавляться постепенно.

## Файлы

- `styles/design-tokens.css` - базовые CSS variables для цветов, поверхностей, текста, отступов, радиусов, теней, анимаций и z-index.
- `styles/main.css` - подключает дизайн-токены после текущих базовых переменных.
- `docs/02-architecture/ui/UI_AUDIT_AND_MODERNIZATION_PLAN.md` - аудит текущего UI и phased rollout.

## Визуальный Тон

Интерфейс должен быть:

- темным, но читаемым;
- спокойным и не кислотным;
- атмосферным, но не декоративно-игровым;
- рабочим инструментом, а не RPG-менюшкой;
- плотным там, где пользователь работает долго;
- мягким в popup и вторичных поверхностях;
- предсказуемым в hover/focus/active состояниях.

## Design Tokens

Новые стили должны сначала искать подходящий токен в `styles/design-tokens.css`.

Основные группы:

- `--color-bg-*` - уровни фона приложения, панелей, карточек и elevated-поверхностей.
- `--color-text-*` - основной, приглушенный и слабый текст.
- `--color-accent*` - пастельно-желтый акцент для focus, hover и выбранных состояний.
- `--color-danger*` - мягкий красный для удаления, закрытия и опасных действий.
- `--color-border-*` - спокойные границы и hover-границы.
- `--space-*` - единая шкала отступов.
- `--radius-*` - радиусы для контролов, панелей и popup.
- `--shadow-*` - тени панелей и popup.
- `--motion-*` и `--ease-standard` - длительность и easing микродвижений.
- `--z-*` - z-index для dropdown, popup, modal и toast.

## Компонентные Правила

### Кнопки

Кнопка должна иметь состояния:

- default;
- hover;
- active;
- focus-visible;
- disabled;
- danger, если действие удаляет или закрывает.

Нельзя добавлять ярко-синий default focus/checkbox/accent. Основной акцент проекта - пастельно-желтый. Опасный акцент - пастельно-красный.

### Inputs / Selects

Поля должны:

- иметь видимый focus;
- не вылезать за label;
- сохранять читаемость на узких окнах;
- не использовать системный синий акцент;
- иметь мягкую границу и фон поверх текущей панели.

### Popup

Popup должен:

- использовать общий popup-position / boundary logic;
- помещаться в viewport;
- закрываться повторным нажатием на кнопку-триггер, Escape или outside click, если это не ломает сценарий;
- иметь мягкое появление только если это не мешает работе;
- не использовать тяжелый blur на больших поверхностях.

### Карточки И Блоки

Карточка и блоки должны:

- сохранять плотную рабочую структуру;
- не превращаться в landing-page;
- иметь понятный заголовок, actions и содержимое;
- использовать runtime controls только как runtime UI;
- не смешивать persistent content и декоративные controls.

### Карта

Карта важнее по производительности, чем по анимациям.

Разрешено:

- hover/focus на маленьких controls;
- мягкое открытие маленьких popup;
- подсветка выбранных токенов/фигур;
- визуальная обратная связь drag/select.

Нельзя:

- добавлять heavy blur на stage;
- анимировать большие слои карты при drag/pan/zoom;
- добавлять долгие transitions на fog/tokens/shapes;
- менять z-index без проверки презентации.

## Animation Rules

Разрешено:

- мягкое открытие popup;
- hover/focus states;
- раскрытие панелей;
- смена активного элемента;
- появление карточек;
- плавные подсветки;
- микроанимации кнопок;
- transition для toolbar/popup.

Запрещено:

- долгие анимации;
- анимации, мешающие работе;
- анимации при drag карты, если они ухудшают производительность;
- heavy blur на больших поверхностях;
- анимации, которые игнорируют `prefers-reduced-motion`.

Обязательное правило reduced motion уже заложено в `styles/design-tokens.css`.

## Accessibility

Минимум для новых UI-элементов:

- видимый `:focus-visible`;
- понятный `title` или текст для иконок;
- достаточный contrast текста;
- не полагаться только на цвет для опасных действий;
- не скрывать текст обрезкой, если он важен для выбора.

## Что Нельзя Делать

- Нельзя переписывать весь дизайн одним большим PR/коммитом.
- Нельзя хардкодить новые цвета, если уже есть подходящий токен.
- Нельзя возвращать синий системный accent для checkbox/select/focus.
- Нельзя добавлять анимации на карту без performance smoke.
- Нельзя менять popup-поведение без popup lifecycle/browser tests.
- Нельзя делать UI “игровой менюшкой” вместо рабочего инструмента мастера.

## Как Добавлять Новый UI

1. Проверить этот контракт.
2. Выбрать существующие design tokens.
3. Если токена не хватает, добавить его в `styles/design-tokens.css` и описать причину.
4. Проверить hover/focus/active/disabled.
5. Проверить popup boundary, если есть popup.
6. Запустить `npm run verify`.
7. Запустить `npm run test:browser`, если меняется UI/runtime.
