# Properties System Contract

Дата обновления: 11.06.2026

## Назначение

`Свойства` становятся главным пользовательским местом, где карточка хранит игровые параметры: стоимость предмета, урон заклинания, дальность навыка, хиты персонажа, КЗ, состояния, эффекты, ручные заметки и будущие расчетные поля.

Главная цель системы: человек сначала видит простой и понятный блок, а глубина появляется только по запросу. Пользователь не должен разбираться в отдельной сущности правил, JSON или внутренних ключах, чтобы добавить поле `Эффект`, изменить хиты или вручную поправить рассчитанное значение.

## Продуктовый Принцип

Интерфейс идет от простого к глубокому:

1. В карточке виден обычный блок `Свойства` с базовыми полями для текущего типа карточки.
2. Если пользователю нужно больше, он нажимает шестеренку.
3. В настройках можно добавить параметр, поменять видимость поля, выбрать тип значения и включить ручное значение.
4. Сложные вещи вроде расчетов, источников эффектов, layout-сетки и миграции старых блоков спрятаны в глубину и не перегружают основной список блоков.

## Чего Нельзя Делать

- Нельзя превращать главное меню блоков в длинный список специальных блоков.
- Нельзя заставлять пользователя создавать отдельную сущность `Rule Tree`, чтобы добавить простой параметр.
- Нельзя рассчитывать данные по русским подписям в HTML. Расчеты читают стабильные ключи модели.
- Нельзя удалять старые блоки резко: существующие карточки должны открываться.
- Нельзя делать автоматическую миграцию, которая переписывает карточку без понятного пользовательского действия или безопасного fallback.

## Пользовательский UI

### Блок Свойства

Блок `Свойства` должен выглядеть как компактная панель полей:

- заголовок `Свойства`;
- базовые поля текущего типа карточки;
- мягкая кнопка-шестеренка;
- визуальная подсветка ручных значений;
- понятные подписи вроде `Факт`, `Макс`, `Временные`, если поле состоит из нескольких частей.

### Шестеренка

Шестеренка открывает popup настроек блока. Popup не должен быть похож на техническую админку.

В popup должны быть:

- список параметров;
- кнопка `Добавить параметр`;
- выбор из понятных готовых параметров;
- возможность создать свой параметр;
- выбор типа значения;
- переключатель видимости поля;
- безопасная кнопка удаления параметра.

Глубокие настройки открываются внутри popup постепенно. На первом экране не должно быть много технических вариантов.

### Пользовательские Параметры

Пользователь может добавить поле, которого нет в базовой схеме типа карточки.

Пример: у типа `Предмет` базово есть стоимость и эффект, но если нужен `Радиус`, пользователь добавляет это поле вручную. При этом карточка не меняет тип, а поле сохраняется как пользовательское.

Текущий MVP поддерживает пользовательские параметры типов:

- `text` - короткий текст;
- `number` - число;
- `textarea` - длинный текст;
- `checkbox` - да/нет.

Эти поля сохраняются внутри persistent HTML блока `Свойства` с признаками `data-property-custom="true"` и читаются `PropertiesModel` как `customFields` и `customValues`. Расширенные типы `select`, `entity`, `entity-list`, расчетные поля и визуальный layout editor остаются следующими слоями.

## Типы Полей

Минимальный набор типов:

- `text` - короткий текст;
- `textarea` - длинный текст;
- `number` - число;
- `select` - выбор из списка;
- `checkbox` - да/нет;
- `entity` - ссылка на карточку;
- `entity-list` - список ссылок на карточки;
- `compound` - группа полей, например хиты `факт / макс / временные`;
- `calculated` - рассчитанное поле с возможностью ручного override.

## Базовые Схемы

Каждый тип карточки получает базовую схему. Она нужна только как стартовая подсказка, а не как жесткая клетка.

Примеры:

- `Предмет`: стоимость, эффект, вес, редкость.
- `Магия`: урон, дальность, размер области, форма, эффект, тип действия.
- `Навык`: уровень навыка, действие, дальность, размер, форма, эффект.
- `Персонаж`: хиты, КЗ, скорость, характеристики, спасброски, навыки, состояния, эффекты.
- `Существо`: как персонаж, но с упрощенным стартовым набором.
- `Объект`: размер, прочность, эффект, изображение.
- `Локация` и `Регион`: тип местности, опасность, описание, связанные сущности.

Пользователь может добавить поле из другой схемы, если это помогает его миру.

## Persistent Model

Данные свойств должны храниться как model-first JSON внутри блока `Свойства`, а не извлекаться из подписи в HTML.

Целевая структура:

```js
{
  kind: 'PropertiesModel',
  version: 2,
  cardType: 'character',
  layoutVersion: 1,
  fields: [
    {
      id: 'field-hp-current',
      key: 'hpCurrent',
      label: 'Хиты',
      type: 'number',
      value: 10,
      source: 'base',
      visible: true,
      layout: {
        x: 0,
        y: 0,
        w: 1,
        h: 1
      },
      calculation: null,
      override: {
        enabled: false,
        value: null
      }
    }
  ],
  customFields: [],
  legacySources: []
}
```

## Runtime UI

Runtime UI не сохраняется как контент карточки.

Runtime-элементы:

- шестеренка;
- popup настроек;
- кнопки добавления и удаления параметров;
- drag handles;
- resize handles;
- подсказки;
- временные ошибки валидации.

Все runtime-элементы получают `data-runtime="true"` или живут вне persistent HTML блока.

## Ручные Значения

Любое рассчитанное поле может быть изменено вручную.

Правила:

1. Если пользователь вручную изменил рассчитанное поле, включается `override.enabled = true`.
2. Поле подсвечивается светло-фиолетовым.
3. Расчетный backend больше не перезаписывает значение поля.
4. Если пользователь очищает ручное значение, `override.enabled = false`.
5. После очистки авторасчет снова включается.

Это правило обязательно для характеристик, модификаторов, хитов, КЗ, скорости, навыков и будущих расчетных параметров.

## Расчетный Backend

Расчеты должны жить отдельно от DOM.

Слой расчетов читает:

- `PropertiesModel`;
- `CharacterModel`;
- legacy-источники только как fallback;
- будущие внешние источники через явные provider-слои.

Расчеты возвращают результат и причину:

```js
{
  key: 'initiative',
  value: 2,
  source: 'calculated',
  formula: 'dexModifier + effects.initiative',
  parts: [
    { label: 'ЛОВ', value: 2 },
    { label: 'Эффекты', value: 0 }
  ]
}
```

В интерфейсе причина может показываться позже через подсказку или popup, но базовый экран не должен быть перегружен формулами.

Текущий foundation-файл: `js/properties/propertiesCalculationEngine.js`.

Поддержанные расчёты:

- уровень;
- бонус мастерства;
- модификаторы характеристик;
- инициатива;
- КЗ;
- скорость;
- summary хитов.

Каждый расчет возвращает:

- `value` - итоговое значение;
- `calculatedValue` - значение без ручного override;
- `source` - `calculated` или `manual`;
- `formula` - человеко-читаемая формула;
- `parts` - составляющие расчёта;
- `override` - состояние ручного значения.

`CharacterModel` получает этот слой как `model.calculations`, но текущие публичные поля `level`, `armorClass`, `speed`, `abilities`, `health` остаются совместимыми.

## Связь С CharacterModel

`CharacterModel` остается backend-слоем для персонажей и существ.

Новый порядок источников:

1. `PropertiesModel` версии 2.
2. `PropertiesModel` версии 1.
3. legacy `Стат. блок DnD`.
4. legacy `Лист персонажа`, если он содержит данные.
5. безопасные defaults.

`Лист персонажа` не должен быть отдельной витриной, которая не редактируется. Целевой путь: лист является режимом отображения/редактирования `Свойств` персонажа.

## Legacy Blocks

Старые блоки не удаляются резко.

Legacy-блоки:

- `Стат. блок DnD`;
- `Стат. блок DnD v2`;
- `Лист персонажа`;
- `Состояния и эффекты`;
- `Предметы`;
- `Заклинания`;
- `Навыки`;
- старый экспериментальный блок `Переменные`.

Целевое поведение:

1. Старые карточки открываются без поломки.
2. Новый код умеет читать их как fallback.
3. В будущем может появиться кнопка `Перенести в Свойства`.
4. Автоматическая миграция разрешена только после отдельного плана, тестов и backup/recovery проверки.

## Упрощение Блоков

Главный список блоков должен стремиться к короткому набору:

- `Текстовый блок`;
- `Блок списка`;
- `Таблица`;
- `Картинка`;
- `Свойства`.

Специализированные сценарии должны уходить внутрь режимов:

- предметы, заклинания и навыки становятся режимами `Блока списка`;
- статы, состояния, эффекты и лист персонажа становятся режимами `Свойств`;
- дополнительные параметры появляются через шестеренку.

## Блок Списка

`Блок списка` является универсальным контейнером для ссылок на карточки.

Внутри блока есть селектор режима:

- предметы;
- заклинания;
- персонажи;
- существа;
- навыки;
- объекты;
- будущие типы.

Режим меняет picker и подписи, но не создает новый тип блока.

## Layout Полей

Поля `Свойств` должны получить собственный layout.

Минимальная модель:

- `x`, `y` - позиция в сетке;
- `w`, `h` - размер в сетке;
- `order` - fallback-порядок для старых layout;
- `collapsed` - свернуто ли поле или группа;
- `groupId` - принадлежность к группе.

На первом этапе можно начать без визуального drag/resize, но контракт должен сразу учитывать будущую динамическую сетку.

## Safe Save

Сохранение должно:

1. Синхронизировать значения inputs/select/checkbox в JSON.
2. Удалять runtime UI.
3. Пропускать HTML через sanitizer.
4. Не сохранять popup, handles, подсказки и временные состояния.
5. Сохранять модель даже если визуальная сетка временно не отрендерилась.

## Файлы

Целевые зоны:

- `js/properties/` - model, schemas, calculations, user fields, layout.
- `js/editor/blocks/` - runtime UI блока, popup настроек, controls.
- `js/templates/blockTypes.js` - генерация persistent HTML блока `Свойства`.
- `js/templates/propertyBlockDefinitions.js` - совместимость со старыми схемами.
- `docs/02-architecture/contracts/PROPERTIES_MODEL_CONTRACT.md` - этот контракт.
- `docs/02-architecture/contracts/CHARACTER_MODEL_CONTRACT.md` - связь с персонажем и расчетами.

## Тесты

Для P0/P1 изменений нужны проверки:

- создание блока `Свойства`;
- открытие шестеренки;
- добавление пользовательского параметра;
- сохранение и повторное открытие параметра;
- ручной override расчетного поля;
- очистка override и возврат к авторасчету;
- добавление поля из другой схемы;
- открытие старой карточки с legacy `Стат. блок DnD`;
- открытие старой карточки с legacy `Состояния и эффекты`;
- проверка, что runtime UI не попадает в сохраненный HTML.

## План Внедрения

1. Закрыть контракт и договориться о модели.
2. Добавить шестеренку и пустой popup настроек.
3. Добавить пользовательские параметры без расчетов: **сделано MVP**.
4. Добавить manual override для calculated fields: **сделано foundation backend**.
5. Подключить CharacterModel к новой модели свойств.
6. Сделать editable character sheet как режим `Свойств`.
7. Упростить список блоков.
8. Перевести item/spell/skill blocks в универсальный `Блок списка`.
9. Добавить layout editor.
10. Подготовить миграцию legacy-блоков.
