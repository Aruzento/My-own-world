# Properties System Contract

Дата обновления: 20.07.2026

## Назначение

`Свойства` становятся главным пользовательским местом, где карточка хранит игровые параметры: стоимость предмета, урон заклинания, дальность навыка, хиты персонажа, КЗ, состояния, эффекты, ручные заметки и будущие расчетные поля.

Главная цель системы: человек сначала видит простой и понятный блок, а глубина появляется только по запросу. Пользователь не должен разбираться в отдельной сущности правил, JSON или внутренних ключах, чтобы добавить поле `Эффект`, изменить хиты или вручную поправить рассчитанное значение.

## Продуктовый Принцип

Интерфейс идет от простого к глубокому:

1. В карточке виден обычный блок `Свойства` с базовыми полями для текущего типа карточки.
2. Если пользователю нужно больше, он нажимает шестеренку.
3. В настройках можно добавить параметр, поменять видимость поля, выбрать тип значения и включить ручное значение.
4. Сложные вещи вроде расчетов, источников эффектов, layout-сетки и миграции старых блоков спрятаны в глубину и не перегружают основной список блоков.

## Чего Нельзя Делать

- Нельзя превращать главное меню блоков в длинный список специальных блоков.
- Нельзя заставлять пользователя создавать отдельную сущность `Rule Tree`, чтобы добавить простой параметр.
- Нельзя рассчитывать данные по русским подписям в HTML. Расчеты читают стабильные ключи модели.
- Нельзя удалять старые блоки резко: существующие карточки должны открываться.
- Нельзя делать автоматическую миграцию, которая переписывает карточку без понятного пользовательского действия или безопасного fallback.

## Пользовательский UI

### Блок Свойства

Блок `Свойства` должен выглядеть как компактная панель полей:

- заголовок `Свойства`;
- базовые поля текущего типа карточки;
- мягкая кнопка-шестеренка;
- визуальная подсветка ручных значений;
- понятные подписи вроде `Факт`, `Макс`, `Временные`, если поле состоит из нескольких частей.

### Шестеренка

Шестеренка открывает popup настроек блока. Popup не должен быть похож на техническую админку.

В popup должны быть:

- список параметров;
- кнопка `Добавить параметр`;
- выбор из понятных готовых параметров;
- возможность создать свой параметр;
- выбор типа значения;
- переключатель видимости поля;
- безопасная кнопка удаления параметра.

Глубокие настройки открываются внутри popup постепенно. На первом экране не должно быть много технических вариантов.

### Пользовательские Параметры

Пользователь может добавить поле, которого нет в базовой схеме типа карточки.

Пример: у типа `Предмет` базово есть стоимость и эффект, но если нужен `Радиус`, пользователь добавляет это поле вручную. При этом карточка не меняет тип, а поле сохраняется как пользовательское.

Текущий MVP поддерживает пользовательские параметры типов:

- `text` - короткий текст;
- `number` - число;
- `textarea` - длинный текст;
- `checkbox` - да/нет.

Эти поля сохраняются внутри persistent HTML блока `Свойства` с признаками `data-property-custom="true"` и читаются `PropertiesModel` как `customFields` и `customValues`. Расчетные override сохраняются как hidden custom-поля `override-*`, чтобы `PropertiesCalculationModel` мог отличать ручное значение от автоматического.

## Типы Полей

Минимальный набор типов:

- `text` - короткий текст;
- `textarea` - длинный текст;
- `number` - число;
- `select` - выбор из списка;
- `checkbox` - да/нет;
- `entity` - ссылка на карточку;
- `entity-list` - список ссылок на карточки;
- `compound` - группа полей, например хиты `факт / макс / временные`;
- `skillGroup` - мульти-поле DnD-навыков и спасбросков одной характеристики;
- `calculated` - рассчитанное поле с возможностью ручного override.

## Базовые Схемы

Каждый тип карточки получает базовую схему. Она нужна только как стартовая подсказка, а не как жесткая клетка.

Примеры:

- `Предмет`: стоимость, эффект, вес, редкость.
- `Магия`: урон, дальность, размер области, форма, эффект, тип действия.
- `Навык`: уровень навыка, действие, дальность, размер, форма, эффект.
- `Персонаж`: уровень, бонус мастерства, инициатива, хиты, КЗ, скорость, доспех, характеристики, спасброски и навыки.
- `Существо`: как персонаж, но с упрощенным стартовым набором.
- `Объект`: размер, прочность, эффект, изображение.
- `Локация` и `Регион`: тип местности, опасность, описание, связанные сущности.

Пользователь может добавить поле из другой схемы, если это помогает его миру.

Для `Персонаж` и `Существо` DnD-навыки хранятся как шесть широких мульти-полей:

- `Навыки СИЛ`: спасбросок СИЛ и Атлетика;
- `Навыки ЛОВ`: спасбросок ЛОВ, Акробатика, Ловкость рук, Скрытность;
- `Навыки ТЛС`: спасбросок ТЛС;
- `Навыки ИНТ`: спасбросок ИНТ, Анализ, История, Магия, Природа, Религия;
- `Навыки МДР`: спасбросок МДР, Внимательность, Выживание, Медицина, Проницательность, Уход за животными;
- `Навыки ХАР`: спасбросок ХАР, Выступление, Запугивание, Обман, Убеждение.

Каждая строка внутри `skillGroup` дает числовое поле результата и checkbox владения. Стабильные ключи сохраняются не по русским подписям, а как `skillAthletics`, `skillStealth`, `saveDex`, `skillAthleticsProficient`, `saveDexProficient` и т.д.

Авторасчет навыков уже включен в runtime блока `Свойства` и в `PropertiesCalculationModel`: значение = модификатор связанной характеристики + бонус мастерства, если владение включено, или + двойной бонус мастерства, если включена экспертность. Если пользователь вручную меняет числовое поле навыка, оно становится manual override и подсвечивается; очистка поля возвращает авторасчет.

`Бонус мастерства` (`proficiencyBonus`) и `Инициатива` (`initiative`) являются видимыми стандартными полями персонажа/существа. `proficiencyBonus` считается от уровня, `initiative` считается от модификатора ЛОВ. Оба поля можно изменить вручную; ручное значение не перетирается runtime-авторасчетом и передается в `CharacterModel` через `manualOverrides`.

КЗ персонажа/существа считается через поле `Доспех` (`armorItem`) и свойства выбранной карточки-предмета. У предмета для этого есть поля `Тип доспеха`, `Базовая КЗ доспеха`, `Лимит ЛОВ к КЗ`. Поле `armorItem` в runtime UI должно предлагать только карточки типа `item`, у которых собственный блок `Свойства` содержит `Тип доспеха`, отличный от `Нет`/пустого значения. Старая сохраненная ссылка на обычный предмет может быть показана как недоступная, но не должна участвовать в расчете КЗ. Правила DnD:

- без доспеха: `10 + модификатор ЛОВ`;
- легкий доспех: `base + модификатор ЛОВ`;
- средний доспех: `base + min(модификатор ЛОВ, лимит или 2)`;
- тяжелый доспех: `base`;
- щит: `10 + модификатор ЛОВ + base`.

Если доспех не выбран, но в старой карточке уже явно записана КЗ, значение считается legacy/manual базой для совместимости.

## Compound Fields

`Properties` may expose a visible compound field when several stable model keys are one human concept.

Current example: item armor settings are one visible `Armor` field (`armorProfile`) with three nested controls:

- `armorKind` - armor type;
- `armorBaseAc` - armor AC/base AC;
- `armorDexMax` - DEX modifier limit.

The compound field is a UI/layout grouping only. `PropertiesModel` and calculation code still read the nested stable keys, so old calculations, `armorItem` filtering and `CharacterModel` do not need a parallel armor model.

Rules:

- a compound field can be deleted/restored from the gear popup as one row;
- nested controls must keep their own `data-property-name` values;
- `getSchemaValueFields()` must flatten compound fields for model reads;
- field layout belongs to the compound wrapper, not to each nested control;
- use compound fields only when deleting or moving the parts separately would be confusing for the owner.

## Layout Grid

Стартовая раскладка персонажа/существа должна повторять пользовательский эталон, принятый 20.07.2026: верхняя строка содержит компактные боевые поля (`Уровень`, `БМ`, `Инициатива`, `КЗ`, `Скорость`, `Доспех`, HP), характеристики идут одной строкой в порядке `СИЛ`, `ЛОВ`, `ТЛС`, `ИНТ`, `МДР`, `ХАР`, а группы навыков стоят широкими блоками ниже, чтобы подписи оставались читаемыми. Эта раскладка является стартовым default layout для новых блоков, но уже созданные карточки не мигрируются автоматически.

Блок `Свойства` использует свободную 12-колоночную layout-сетку. Положение поля хранится как координаты `x/y` и размер `w/h`; порядок DOM остается только fallback-метаданными и не должен автоматически перепаковывать поля.

Правила:

- drag поля записывает координаты клетки под курсором, а не индекс между соседними полями;
- если целевая область занята, существующие поля сдвигаются вниз до ближайшего свободного пространства;
- пустые строки и пустые клетки допустимы и должны сохраняться после save/reload;
- `synchronizePropertyBlockLayout` нормализует существующий layout, но не стягивает поля к последнему заполненному месту;
- resize слева или сверху меняет соответствующую сторону поля через `x/y/w/h`, а не переставляет поле в DOM;
- новые пользовательские поля получают ближайшее свободное место в сетке;
- стартовый layout персонажа и существа должен быть человеко-понятным: компактные боевые числа сверху, HP отдельной строкой, шесть характеристик на одной строке по 2 колонки.

Для персонажей стандартные маленькие поля (`Уровень`, `БМ`, `Инициатива`, `КЗ`, `Скорость`, временные HP) занимают компактные слоты, HP занимает средние слоты, а `СИЛ/ЛОВ/ТЛС/ИНТ/МДР/ХАР` обязаны помещаться на одной строке без переноса.

## Persistent Model

Данные свойств должны храниться как model-first JSON внутри блока `Свойства`, а не извлекаться из подписи в HTML.

Целевая структура:

```js
{
  kind: 'PropertiesModel',
  version: 2,
  cardType: 'character',
  layoutVersion: 1,
  fields: [
    {
      id: 'field-hp-current',
      key: 'hpCurrent',
      label: 'Хиты',
      type: 'number',
      value: 10,
      source: 'base',
      visible: true,
      layout: {
        x: 0,
        y: 0,
        w: 1,
        h: 1
      },
      calculation: null,
      override: {
        enabled: false,
        value: null
      }
    }
  ],
  customFields: [],
  manualOverrides: {},
  legacySources: []
}
```

## Runtime UI

Runtime UI не сохраняется как контент карточки.

Runtime-элементы:

- шестеренка;
- popup настроек;
- кнопки добавления и удаления параметров;
- drag handles;
- resize handles;
- бейджи модификаторов характеристик;
- подсказки;
- временные ошибки валидации.

Все runtime-элементы получают `data-runtime="true"` или живут вне persistent HTML блока.

## Ручные Значения

Любое рассчитанное поле может быть изменено вручную.

Правила:

1. Если пользователь вручную изменил рассчитанное поле, включается `override.enabled = true`.
2. Поле подсвечивается светло-фиолетовым.
3. Расчетный backend больше не перезаписывает значение поля.
4. Если пользователь очищает ручное значение, `override.enabled = false`.
5. После очистки авторасчет снова включается.

Это правило обязательно для бонуса мастерства, инициативы, КЗ, навыков и будущих расчетных параметров. Модификаторы характеристик показываются runtime-бейджами рядом с базовым значением характеристики и не сохраняются как отдельное пользовательское поле.

## Расчетный Backend

Расчеты должны жить отдельно от DOM.

Слой расчетов читает:

- `PropertiesModel`;
- `CharacterModel`;
- legacy-источники только как fallback;
- будущие внешние источники через явные provider-слои.

Расчеты возвращают результат и причину:

```js
{
  key: 'initiative',
  value: 2,
  source: 'calculated',
  formula: 'dexModifier + effects.initiative',
  parts: [
    { label: 'ЛОВ', value: 2 },
    { label: 'Эффекты', value: 0 }
  ]
}
```

В интерфейсе причина может показываться позже через подсказку или popup, но базовый экран не должен быть перегружен формулами.

Текущий foundation-файл: `js/properties/propertiesCalculationEngine.js`.

Поддержанные расчёты:

- уровень;
- бонус мастерства;
- модификаторы характеристик;
- инициатива;
- КЗ;
- скорость;
- summary хитов.

Каждый расчет возвращает:

- `value` - итоговое значение;
- `calculatedValue` - значение без ручного override;
- `source` - `calculated` или `manual`;
- `formula` - человеко-читаемая формула;
- `parts` - составляющие расчёта;
- `override` - состояние ручного значения.

Видимые runtime-поля в блоке `Свойства` должны совпадать с расчетным backend: изменение уровня обновляет `БМ`, изменение характеристики обновляет модификатор, связанные навыки и инициативу, а ручной override сохраняет пользовательскую правку.

`CharacterModel` получает этот слой как `model.calculations`, но текущие публичные поля `level`, `armorClass`, `speed`, `abilities`, `health` остаются совместимыми.

## Связь С CharacterModel

`CharacterModel` остается backend-слоем для персонажей и существ.

Новый порядок источников:

1. `PropertiesModel` версии 2.
2. `PropertiesModel` версии 1.
3. legacy `Стат. блок DnD`.
4. legacy `Лист персонажа`, если он содержит данные.

Старые блоки `Статистика персонажа` и `Стат. блок DnD` не должны предлагаться в popup `Добавить блок` для новых карточек. Они остаются только как совместимость чтения, upgrade и fallback для уже сохраненных `.md`.
5. безопасные defaults.

`Лист персонажа` не должен быть отдельной витриной, которая не редактируется. Текущий MVP: лист редактирует значения через блок `Свойства`, при необходимости создает этот блок на карточке и вызывает сохранение страницы. Сам лист остается runtime UI и не сохраняет собственные игровые числа.

## Legacy Blocks

Старые блоки не удаляются резко.

Legacy-блоки:

- `Стат. блок DnD`;
- `Стат. блок DnD v2`;
- `Лист персонажа`;
- `Состояния и эффекты`;
- `Предметы`;
- `Заклинания`;
- `Навыки`;
- старый экспериментальный блок `Переменные`.

Целевое поведение:

1. Старые карточки открываются без поломки.
2. Новый код умеет читать их как fallback.
3. `propertiesLegacyBridge` распознает legacy-блоки и возвращает отчет для будущего UI.
4. В будущем может появиться кнопка `Перенести в Свойства`.
5. Автоматическая миграция разрешена только после отдельного плана, тестов и backup/recovery проверки.

## Упрощение Блоков

Главный список блоков должен стремиться к короткому набору:

- `Текстовый блок`;
- `Блок списка`;
- `Таблица`;
- `Картинка`;
- `Свойства`.

Специализированные сценарии должны уходить внутрь режимов:

- предметы, заклинания и навыки становятся режимами `Блока списка`;
- статы, состояния, эффекты и лист персонажа становятся режимами `Свойств`;
- дополнительные параметры появляются через шестеренку.

## Блок Списка

`Блок списка` является универсальным контейнером для ссылок на карточки.

Внутри блока есть селектор режима:

- предметы;
- заклинания;
- персонажи;
- существа;
- навыки;
- объекты;
- будущие типы.

Режим меняет picker и подписи, но не создает новый тип блока.

Текущий MVP использует `data-block-type="list"` и сохраняет выбранный режим в `data-list-kind`. Для совместимости он переиспользует существующие chip-классы старых item/spell/skill-блоков, но пользователь видит один универсальный блок.

## Зависимые Переменные

`CardVariablesModel` превращает поля блока `Свойства` в переменные карточки. Foundation `cardVariableDependencies.js` добавляет поверх этого зависимые расчеты между карточками.

Пример:

- у персонажа есть пользовательское поле `race = human`;
- карточка `human` содержит собственные `Свойства`, например `str = 1`;
- формула `self.str + race.str + 2` может быть рассчитана как безопасная сумма с понятными частями расчета.

Правила:

1. расчет зависимостей не использует `eval` и не исполняет произвольный JS;
2. зависимости ищут карточки по `id`, `title` или `aliases`;
3. результат содержит diagnostics, если переменная или карточка не найдена;
4. текущий foundation не добавляет визуальный конструктор формул и не записывает результаты обратно в HTML;
5. карта будет учитывать такие зависимости только после явного подключения через `CharacterModel`.

## Layout Полей

Поля `Свойств` должны получить собственный layout.

Минимальная модель:

- `x`, `y` - позиция в сетке;
- `w`, `h` - размер в сетке;
- `order` - fallback-порядок для старых layout;
- `collapsed` - свернуто ли поле или группа;
- `groupId` - принадлежность к группе.

Текущий MVP:

- каждое поле получает встроенный runtime grip-handle;
- перестановка полей меняет порядок persistent HTML;
- layout поля хранится в persistent `data-property-layout` как модель `x/y/w/h/order/collapsed/groupId`;
- для обратной совместимости `data-property-span` и `data-property-rows` остаются производными от layout-модели;
- ширина меняется по 12-колоночной сетке, а высота - по шагам строк;
- поле можно растягивать за стороны и углы через восемь runtime resize-точек;
- при растягивании за левую или верхнюю сторону двигается именно эта сторона поля, а не противоположная;
- drag можно завершить не только над другим полем, но и в пустой области сетки блока;
- runtime grip и resize-точки не являются `<button>`, чтобы клик по input/textarea внутри label не активировал служебный control и не мешал вводу текста;
- во время drag появляется runtime-placeholder, который показывает будущую позицию поля и мягко сдвигает соседей;
- реальное поле во время drag не переставляется на каждый `pointermove`: оно временно скрывается, под курсором идет runtime ghost, а persistent DOM-порядок меняется один раз при drop;
- placeholder двигается через кадрированный `requestAnimationFrame`-цикл и FLIP-анимацию, чтобы сетка не дергалась от частых pointer-событий;
- popup шестеренки показывает безопасную кнопку удаления у каждого видимого поля, включая базовые поля схемы;
- popup шестеренки не должен содержать отдельную кнопку размера: размер меняется только напрямую через resize-точки поля;
- добавление поля предлагает готовые параметры из существующих схем и расчетного слоя, чтобы пользователь не вспоминал точные названия вручную;
- `PropertiesModel` читает layout каждого базового и пользовательского поля в `model.layout`, а custom-поля дополнительно несут свой `layout` рядом со значением;
- группировки через `groupId` и свернутость через `collapsed` уже зарезервированы в модели, но пользовательский UI групп еще не включен.

## Safe Save

Сохранение должно:

1. Синхронизировать значения inputs/select/checkbox в JSON.
2. Удалять runtime UI.
3. Пропускать HTML через sanitizer.
4. Не сохранять popup, handles, подсказки и временные состояния.
5. Сохранять модель даже если визуальная сетка временно не отрендерилась.

## Файлы

Целевые зоны:

- `js/properties/` - model, schemas, calculations, user fields, layout.

## Тесты

Для P0/P1 изменений нужны проверки:

- создание блока `Свойства`;
- открытие шестеренки;
- добавление пользовательского параметра;
- сохранение и повторное открытие параметра;
- ручной override расчетного поля;
- очистка override и возврат к авторасчету;
- добавление поля из другой схемы;
- открытие старой карточки с legacy `Стат. блок DnD`;
- открытие старой карточки с legacy `Состояния и эффекты`;
- проверка, что runtime UI не попадает в сохраненный HTML.
