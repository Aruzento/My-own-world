# Properties Model Contract

Дата обновления: 01.06.2026

## Назначение

`PropertiesModel` — это первый model-first слой для карточек, которым нужен набор типизированных игровых свойств: предметы, навыки, магия, существа, объекты, локации и регионы.

Идея старого блока `Переменные` теперь закреплена здесь: **свойства карточки являются переменными этой сущности**. У разных типов карточек разные схемы свойств, но для расчетных подсистем это один принцип: стабильный ключ + тип значения + значение пользователя.

Главная цель: не превращать каждый новый расчет в поиск по HTML и русским подписям. HTML остается пользовательским интерфейсом, а расчетные подсистемы читают стабильные ключи: `hpCurrent`, `hpMax`, `damage`, `range`, `actionType`, `skillLevel` и так далее.

## Файлы

- `js/properties/propertySchemas.js` — единый каталог схем свойств по типам карточек.
- `js/properties/propertiesModel.js` — чтение и нормализация блока `Свойства` в модель.
- `js/properties/cardVariablesModel.js` — единый список переменных карточки, построенный из `PropertiesModel`.
- `js/properties/characterCalculations.js` — расчетный слой DnD: модификаторы, бонус мастерства, проверки и чтение хитов.
- `js/templates/propertyBlockDefinitions.js` — совместимый экспорт для старого кода шаблонов.
- `js/templates/blockTypes.js` — рендерит блок `Свойства` из схемы, а не из ручных массивов.

## Правила

1. Новый тип карточки получает схему в `propertySchemas.js`.
2. Поле свойства получает стабильный английский ключ в `name`.
3. Пользовательская подпись может быть русской и может меняться, ключ менять нельзя без миграции.
4. Расчеты не должны искать значения по тексту label.
5. Расчеты читают `PropertiesModel`, legacy `Стат. блок DnD` или будущую character-модель через `characterCalculations.js`.
6. Если тип карточки не поддерживает схему, блок `Свойства` для него не предлагается.
7. Архивные эксперименты `DnD v2` и `Переменные` не мигрируются автоматически до появления полноценного `CharacterModel`.
8. Переменные сущности создаются не отдельным блоком, а через блок `Свойства`.
9. Если расчету нужен список переменных карточки, он должен читать `CardVariablesModel`, а не DOM.

## Card Variables

`CardVariablesModel` - тонкий слой поверх `PropertiesModel`.

Он нужен, чтобы будущие расчеты, Rule Tree, World Packages и CharacterModel могли говорить не "найди поле в HTML", а "дай переменную `armorClass`, `damage`, `range`, `effect`".

Пример:

```js
{
  kind: 'CardVariablesModel',
  version: 1,
  pageId: '...',
  cardType: 'item',
  source: 'properties-block',
  variables: [
    {
      key: 'gold',
      label: 'ЗМ',
      type: 'number',
      value: 10,
      rawValue: '10',
      source: 'properties-block'
    }
  ],
  byKey: {
    gold: { ... }
  }
}
```

Правила:

1. `key` берется из `propertySchemas.js` и является стабильным API.
2. `label` можно менять для UX, но нельзя использовать для расчетов.
3. `number`-поля с пустым значением становятся `null`, а не `0`.
4. `text`, `select` и `textarea` остаются строками.
5. Старый блок `Переменные` не возвращается в активный UI: его идея встроена в свойства карточек.

## DnD-Расчеты

Модификатор характеристики рассчитывается по правилу DnD 5e:

- 1 = -5
- 2-3 = -4
- 4-5 = -3
- 6-7 = -2
- 8-9 = -1
- 10-11 = +0
- 12-13 = +1
- 14-15 = +2
- 16-17 = +3
- 18-19 = +4
- 20-21 = +5
- 22-23 = +6
- 24-25 = +7
- 26-27 = +8
- 28-29 = +9
- 30 = +10

Бонус мастерства:

- уровни 1-4 = +2
- уровни 5-8 = +3
- уровни 9-12 = +4
- уровни 13-16 = +5
- уровни 17-20 = +6

## Что Уже Подключено

Карта при чтении и изменении хитов теперь идет через расчетный слой:

1. Сначала ищется `PropertiesModel` у существа/персонажа.
2. Если его нет, используется legacy `Стат. блок DnD`.
3. Если нет и legacy-блока, существующий код карты создает legacy-блок с хитами 10/10, чтобы не ломать старые сценарии.

## Что Следует Дальше

Следующий крупный шаг — `CharacterModel`: отдельная модель персонажа, которая сможет объединять свойства, статблок, навыки, инвентарь, классы, расы и будущие расчетные источники без привязки к HTML.
