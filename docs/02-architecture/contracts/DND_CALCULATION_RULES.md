# DnD Calculation Rules

Этот файл описывает правила расчетов, которые MyOwnWorld выполняет внутри программы.

Назначение файла:

- дать владельцу продукта понятное место для аудита формул;
- отделить игровые формулы от UI-блоков;
- не искать правила расчета по разным `.js` файлам;
- фиксировать, какие расчеты уже поддержаны, а какие только запланированы.

Важно: этот файл не является runtime-данными workspace. Программа не читает его автоматически. Это контракт для разработки и проверки.

## Главный Принцип

Правила расчета живут в коде приложения, а не в карточках и не в тексте Rule Tree.

Пользовательские карточки и блок `Свойства` дают входные данные:

- характеристики;
- уровень;
- владение навыком;
- доспех;
- хиты;
- эффекты;
- ручные override.

Расчетный слой программы превращает эти данные в итоговые значения:

- модификаторы характеристик;
- бонус мастерства;
- навыки и спасброски;
- КЗ;
- скорость;
- инициативу;
- состояние здоровья для карты.

Если пользователь вручную меняет расчетное поле, ручное значение имеет приоритет. Очистка ручного значения возвращает авторасчет.

## Термины

`score` - значение характеристики, например СИЛ 16.

`modifier` - модификатор характеристики, например для 16 это `+3`.

`proficiencyBonus` - бонус мастерства.

`override` - ручное значение, которое временно сильнее авторасчета.

`effect` - модификатор из состояния, предмета, правила или будущего World Package.

`source` - источник значения: `calculated`, `manual`, `legacy`, `effect`.

## Модификатор Характеристики

Формула:

```text
modifier = floor((score - 10) / 2)
```

Таблица:

| Значение характеристики | Модификатор |
| --- | --- |
| 1 | -5 |
| 2-3 | -4 |
| 4-5 | -3 |
| 6-7 | -2 |
| 8-9 | -1 |
| 10-11 | +0 |
| 12-13 | +1 |
| 14-15 | +2 |
| 16-17 | +3 |
| 18-19 | +4 |
| 20-21 | +5 |
| 22-23 | +6 |
| 24-25 | +7 |
| 26-27 | +8 |
| 28-29 | +9 |
| 30 | +10 |

Текущая реализация:

- `js/properties/characterCalculations.js`;
- `js/properties/propertiesCalculationEngine.js`;
- `js/editor/propertiesAutoCalculations.js`.

## Бонус Мастерства

Формула по уровню DnD 5e:

| Уровень | Бонус мастерства |
| --- | --- |
| 1-4 | +2 |
| 5-8 | +3 |
| 9-12 | +4 |
| 13-16 | +5 |
| 17-20 | +6 |

Правило:

- если уровень пустой или некорректный, используется уровень `1`;
- если пользователь вручную задаст отдельный override бонуса мастерства в будущем UI, он должен быть выше авторасчета.

Текущая реализация:

- `calculateProficiencyBonus()` в `js/properties/characterCalculations.js`;
- расчетная модель в `js/properties/propertiesCalculationEngine.js`.

## Навыки И Спасброски

Каждый навык связан с характеристикой.

Формула без владения:

```text
check = abilityModifier
```

Формула с владением:

```text
check = abilityModifier + proficiencyBonus
```

Текущий MVP не поддерживает expertise/double proficiency. Это будущая задача.

`Навыки СИЛ`:

- спасбросок СИЛ;

`Навыки ЛОВ`:

- спасбросок ЛОВ;
- Акробатика;
- Ловкость рук;

`Навыки ТЛС`:

- спасбросок ТЛС.

`Навыки ИНТ`:

- спасбросок ИНТ;
- Анализ;
- История;

`Навыки МДР`:

`Навыки ХАР`:

## Класс Доспеха

Поле персонажа/существа:

- `armorItem` - ссылка на карточку-предмет по `id`, `title` или `alias`.

Поля предмета-доспеха:

- `armorKind` - тип доспеха;
- `armorBaseAc` - базовая КЗ;
- `armorDexMax` - максимум модификатора ЛОВ для среднего доспеха.

Поддержанные типы:

| Тип | Формула |
| --- | --- |
| Нет | `10 + dexModifier` |
| Легкий | `armorBaseAc + dexModifier` |
| Средний | `armorBaseAc + min(dexModifier, armorDexMax или 2)` |
| Тяжелый | `armorBaseAc` |
| Щит | `10 + dexModifier + armorBaseAc` |

Текущее ограничение:

- отдельный щит поверх доспеха пока не поддержан как экипировка;
- сейчас `Щит` считается как отдельный выбранный защитный предмет;
- будущий слой экипировки должен поддержать доспех + щит + эффекты.

Legacy-правило:

- если `armorItem` не выбран, но в старой карточке уже записана `armorClass`, это значение считается legacy/manual базой совместимости.

Текущая реализация:

- `calculateDndArmorClass()` в `js/properties/propertiesCalculationEngine.js`;
- runtime пересчет в `js/editor/propertiesAutoCalculations.js`.

## Хиты

Основные поля:

- `hpCurrent` - текущие хиты;
- `hpMax` - максимум хитов;
- `hpTemp` - временные хиты.

Правило получения урона:

1. Урон сначала уменьшает `hpTemp`.
2. Остаток урона уменьшает `hpCurrent`.
3. `hpCurrent` не должен уходить ниже `0`.

Правило лечения:

1. Лечение увеличивает только `hpCurrent`.
2. Лечение не пополняет `hpTemp`.
3. `hpCurrent` не должен становиться выше `hpMax`.

Текущая реализация:

- `CharacterModel` в `js/character/characterModel.js`;
- карта читает модель через `js/editor/campaignMapCharacterBridge.js`.

## Хиты На Карте

Карта использует процент здоровья:

```text
healthPercent = 100 * hpCurrent / hpMax
```

Правила отображения:

- если блока `Свойства` или HP-данных нет, карта не делает специальных действий;
- если `hpCurrent = 0`, токен считается павшим;
- цвет рамки токена зависит от процента HP;
- будущие изменения должны идти через `CharacterModel`, а не через прямое чтение DOM.

## Инициатива

Текущий базовый расчет:

```text
initiativeModifier = dexModifier + effects.initiative
```

Правило:

- пользователь может вручную ввести результат реального броска d20 в popup инициативы;
- программа не должна заставлять использовать автоматический roll;
- автоматический roll остается удобной кнопкой, но не единственным путем.

## Эффекты

Эффекты могут менять:

- `armorClass`;
- `speed`;
- `initiative`;
- будущие проверки и навыки.

Источники эффектов:

- карточка персонажа;
- предмет из инвентаря;
- заклинание;
- навык;
- внутренний rules workspace;
- будущий World Package.

Правило:

- карта, инициатива и лист персонажа не должны знать конкретный источник эффекта;
- источник обязан передать модельный effect в `CharacterModel`;
- `CharacterModel` объединяет эффекты и отдает итоговые значения.

## Внутренний Rules Workspace

Внутренний rules workspace хранит текст правил, справочники, условия, пакеты и ссылки.

Он не заменяет расчетный код.

Пример:

- в rules workspace есть страница "Бонус мастерства";
- пользователь может открыть ее через wiki-link или через popup шестеренки;
- но само значение бонуса мастерства считает программа по правилам выше.

## Что Нельзя Делать

- Не рассчитывать игровые значения из русских подписей UI.
- Не выполнять пользовательский JavaScript, `eval` или `Function`.
- Не заставлять пользователя создавать Rule Tree для простого изменения поля карточки.
- Не делать runtime auto-fix испорченных данных формул.
- Не перезаписывать ручной override без явного действия пользователя.
