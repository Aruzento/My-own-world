# Asset Lifecycle Contract

Этот документ описывает, как проект должен хранить, показывать, проверять и удалять ассеты.

Ассет - это внешний файл, на который ссылается страница или сущность: портрет карточки, фон карты, PNG-объект на карте, будущая музыка локации и другие медиа.

## Цели

- Не сохранять временные `blob:` URL в `.md`.
- Понимать, какая страница или сущность владеет файлом.
- Уметь находить сломанные ссылки на файлы.
- Уметь находить orphan assets: файлы, на которые больше никто не ссылается.
- Подготовить основу для будущей музыки локаций и плейлистов.

## Типы Assets

### `image`

Обычная картинка внутри карточки или image block.

Примеры владельцев:

- блок картинки;
- rich image внутри карточки;
- будущий вложенный медиа-блок.

### `portrait`

Главное изображение карточки в `media-box`.

Примеры владельцев:

- карточка персонажа;
- карточка существа;
- карточка предмета;
- карточка объекта.

### `mapBackground`

Фоновое изображение карты кампании.

Особенности:

- за ним закрепляется состояние сетки;
- за ним закрепляется состояние тумана войны;
- при смене фона карта должна загрузить именно тот fog/grid state, который относится к выбранному background.

### `mapObjectPng`

PNG-объект на карте с прозрачным фоном.

Особенности:

- объект может иметь ширину, высоту, поворот и координаты;
- в режиме презентации выделение и рамки не показываются;
- ссылка на PNG должна быть стабильной и не зависеть от временного browser URL.

### `audio`

Будущий одиночный аудиофайл.

Примеры владельцев:

- карточка локации;
- карта кампании;
- сцена или боевой энкаунтер.

### `playlist`

Будущий набор аудиофайлов с порядком и настройками воспроизведения.

Плейлист сам по себе не является файлом. Это persistent-структура, которая хранит ссылки на assets типа `audio`.

### `futureMedia`

Резервный тип для будущих медиа: видео, анимированные карты, webp/avif variants, token sprites.

Новый тип нельзя добавлять только строкой в UI. Для него нужно обновить этот contract, serializer, broken asset checker и orphan detection.

## AssetReference

Любая persistent-ссылка на ассет должна постепенно приводиться к единой форме:

```json
{
  "id": "asset-uuid",
  "path": "assets/images/example.png",
  "type": "portrait",
  "owner": {
    "pageId": "page-uuid",
    "entityId": "optional-token-or-block-id",
    "scope": "card"
  },
  "fallback": null,
  "missing": false
}
```

### Поля

- `id` - стабильный идентификатор ассета внутри workspace.
- `path` - относительный путь внутри workspace, который можно сохранить в `.md`.
- `type` - тип ассета из списка выше.
- `owner` - кто использует ассет.
- `fallback` - запасной путь или режим показа, если файл не найден.
- `missing` - runtime-флаг, который выставляет проверка, если файл не найден.

## Правила Сохранения

- В persistent HTML/JSON можно сохранять только относительные workspace-пути или будущий `AssetReference`.
- Нельзя сохранять `blob:`, `data:` для пользовательских файлов, абсолютные локальные пути и временные browser URL.
- Runtime preview может использовать `blob:`, но serializer обязан заменить его на стабильный путь или удалить перед сохранением.
- Если ассет был удален пользователем через UI, persistent-ссылка на него тоже должна быть удалена.
- Если ассет заменен, старая ссылка должна быть отвязана от владельца.

## Правила Загрузки

- Если файл найден, UI показывает ассет штатно.
- Если файл не найден, UI показывает мягкое missing state, но не пытается автоматически чинить путь.
- Runtime auto-fix запрещен: если ссылка сломана, нужно исправлять источник или дать пользователю инструмент восстановления.
- Broken asset checker должен уметь перечислить страницу, поле и путь, где обнаружена проблема.

## Orphan Assets

Orphan asset - это файл в папке assets, на который нет persistent-ссылок.

Orphan detection не должен сразу удалять файлы. Он должен:

1. Собрать все ссылки из карточек, карт, task tracker и будущих систем.
2. Сравнить их с файлами в asset folders.
3. Показать список кандидатов на удаление.
4. Запросить подтверждение пользователя перед удалением.

- `js/editor/campaignMapTokens.js` и связанные map-модули используют изображения существ и объектов на карте.

## Основа Под Музыку Локаций

Музыка локаций должна использовать тот же asset lifecycle, что и изображения.

Первые persistent-атрибуты уже зарезервированы для будущих блоков и моделей:

- `data-audio-asset` - одиночный аудиофайл;
- `data-playlist-asset` - ссылка на persistent-описание плейлиста.

Плейлист должен хранить порядок треков и настройки воспроизведения отдельно от самих аудиофайлов. Каждый трек внутри плейлиста должен ссылаться на `AssetReference` типа `audio`.
