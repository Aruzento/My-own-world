Текущий Техдолг

Архитектурный

  • Карта слишком большая для одного модуля.
  • DOM всё ещё часто выступает моделью данных.
  • Contenteditable остаётся зоной повышенной хрупкости.
  • Serializer стал лучше, но цена ошибки высокая: можно потерять сохранение.

Продуктовый

  • Карта кампании уже похожа на отдельный продукт внутри продукта.
  • Таблицы, изображения, блоки и wiki-links требуют regression-тестов.
  • Перформанс карты зависит от размеров изображений, количества токенов и CSS-эффектов.
  • UI быстро растёт, нужен единый компонентный контракт.

Сохранность данных

  • Локальные файлы удобны, но завязаны на File System Access API.
  • Autosave, ручное Ctrl+S, fog, HP и drag должны оставаться в write queue.
  • Нет полноценной истории изменений и восстановления.
  • Нет версионированной миграции всей модели проекта.

Поддерживаемость

  • Крупные CSS-файлы трудно проверять глазами.
  • Много popup-ов уже унифицировано, но не всё.
  • execCommand изолирован, но пока не заменён.
  • Нет CI, линтера, форматтера и test gate перед push.

Уязвимости и Риски

Web blocker

HTML injection

Markdown хранит HTML, код часто использует innerHTML. Для локального одиночного пользователя это терпимо; для интернета нужен sanitizer и строгая модель документа.

Data loss

Save pipeline

Если runtime UI попадёт в persistent HTML или write queue обойдут напрямую, возможны трудноуловимые баги сохранения.

Permission UX

File System Access

Браузерные permissions требуют user activation и не подходят как основа для публичного web-сервиса.

Assets

Изображения и память

Большие карты и PNG-объекты требуют лимитов, preview-кэша, lazy loading и освобождения blob/object URL.

No auth

Нет пользователей

Для web нужны аккаунты, права доступа, приватность кампаний, backup и экспорт.

Testing

Нет автоматической сетки безопасности

Проект уже достаточно сложный, чтобы ручные проверки перестали быть достаточными.

Интернет-Ресурс: Реально, Но Это Новый Этаж

Что даст web

  • Доступ с разных устройств.
  • Кампании можно шарить игрокам.
  • Можно сделать совместное редактирование и live-презентацию.
  • Появится нормальный профиль пользователя и тарифы.

Что придётся построить

  • Backend API и базу данных вместо прямой записи `.md`.
  • Object storage для картинок и карт.
  • Auth, роли, приватность, permissions.
  • Sanitizer HTML и безопасный renderer.
  • Миграцию существующих workspace в cloud model.
Вывод: делать web прямо сейчас рано. Сначала нужно отделить модель данных от DOM и стабилизировать карту. Иначе backend просто закрепит текущую хрупкость в более дорогой форме.

Desktop App: Вероятно Лучший Следующий Формат

Почему станет стабильнее

  • Нативный доступ к файлам без браузерных permission-ловушек.
  • Проще контролировать версию runtime, кэш, меню, hotkeys.
  • Можно оставить локальную философию: пользователь владеет своими файлами.
  • Проще добавить backup, export/import, crash recovery.

Что не решится само

  • Монолит `campaignMap.js` останется монолитом.
  • DOM как модель данных останется риском.
  • Без тестов приложение будет ломаться так же, только в упаковке.
  • Security всё равно важна, если открывать чужие workspace.
Рекомендация: после стабилизации ядра идти в Tauri/Electron-обёртку. Для текущего проекта desktop app даст больше пользы быстрее, чем полноценный cloud web.

  • Разделить карту.
  • Ввести map data model.
  • Добавить e2e smoke.
  • Починить docs/UTF-8 при необходимости.
2-4 недели

Надёжное локальное приложение

  • Crash-safe save.
  • History/undo для structural actions.
  • Asset cleanup.
  • Performance budget карты.

  • Tauri или Electron.
  • Нативные file dialogs.
  • Backup/export.
  • Автообновления.

Cloud/Web

  • Backend и DB.
  • Auth и sharing.
  • Sanitized document model.
  • Live presentation sync.

Итоговое Решение

Не делать сейчас

  • Не начинать cloud web до отделения данных от DOM.
  • Не добавлять крупные функции в карту, пока она монолитна.
  • Не полагаться только на ручное тестирование.

Делать сейчас

  • Стабилизировать карту и save pipeline.
  • Разделить код по ответственности.
  • Создать автоматическую сетку smoke-тестов.
  • Готовить проект к desktop app как ближайшему серьёзному формату.
Самый здравый путь: сначала сделать крепкое локальное ядро, затем упаковать его как desktop app, и только после этого проектировать web/cloud. Это даст стабильность без преждевременного большого переписывания.
