## Шкала

- **Уровень 1** - отсутствует или хаотично.
- **Уровень 2** - есть частично, но риск остается высоким.
- **Уровень 3** - формализовано и реально используется.
- **Уровень 4** - системно, повторяемо и частично измеримо.
- **Уровень 5** - зрелая масштабируемая практика.
- **Неприменимо** - критерий сейчас не относится к текущему local-first режиму.

## Общий вывод по шкале

Проект находится примерно на **уровне 3** по архитектурной управляемости, тестированию, техдолгу, local-first базовой модели и AI-поддерживаемости.

При этом отдельные зоны остаются на **уровне 1-2**:

- безопасность и sanitizer;
- CI/CD;
- продуктовые метрики;
- performance governance;
- asset lifecycle;
- PageRepository / PageIndex;
- plugin readiness;
- sync/conflict readiness.

Сильная сторона проекта: он уже перестал быть просто набором экранов и функций. В нем появилась инженерная основа: контракты, модели, serializer-слои, тесты, планы, manual, browser smoke и model-first направление.

Слабая сторона: часть систем уже достаточно сложная, но еще не имеет полноценного единого слоя данных, безопасной HTML-границы и автоматического CI, который бы защищал архитектуру без ручного запуска.

# 1. Управление архитектурой

## 1.1 Cloud Native

### Вопрос 1. Насколько продукт независим от конкретного окружения запуска?

**Оценка: уровень 3.**

Продукт уже делится на UI, model, persistence и serializer-слои. Это видно по карте, таск-трекеру, storage и форматированию. Часть логики тестируется вне браузерного UI. Поэтому переносимость уже выше, чем у обычной HTML-страницы.

Ограничение: продукт все еще сильно завязан на browser runtime и File System Access API. Нет `StorageAdapter`, `AssetAdapter`, `EnvironmentConfig`, поэтому уровень 4 пока недостижим.

Доказательства:

- `js/editor/campaignMapModel.js`
- `js/editor/campaignMapStore.js`
- `js/editor/campaignMapDataSerializer.js`
- `js/taskTracker/taskTrackerModel.js`
- `js/storage/workspaceStorage.js`
- `tests/campaignMapModel.test.mjs`
- `tests/taskTrackerModel.test.mjs`

### Вопрос 2. Насколько продукт готов к будущему переходу в desktop app?

**Оценка: уровень 3.**

Desktop-направление архитектурно понятно: нужно заменить storage, file picker, permissions, asset layer, config и release/update flow. Ядро уже частично отделено от DOM, а local-first подход естественно подходит для desktop.

Ограничение: нет Tauri/Electron spike, нет desktop smoke checklist, нет native file dialogs и packaging notes.

Доказательства:

- `docs/PROJECT_STRATEGY_PRESENTATION.html`
- `README.md`
- `js/storage/workspaceStorage.js`
- `js/storage/assetStorage.js`
- `docs/PLANS_AND_TECH_DEBT.md`

# 2. Управление данными

## 2.1 Метаданные

### Вопрос 1. Насколько системно продукт использует метаданные?

**Оценка: уровень 3.**

Сущности имеют `id`, `parent`, `order`, `template`, `type`, `tags`, `aliases`. Дерево строится по parent-связям. Карта использует `data-page-id`, `data-token-id`, `data-source-mode`, `data-grid`, `data-fog-image`.

Доказательства:

- `README.md`
- `js/core/markdown.js`
- `js/tree/treeUtils.js`
- `js/editor/campaignMapDataSerializer.js`
- `js/editor/campaignMapModel.js`

### Вопрос 2. Насколько продукт способен использовать метаданные как движок поведения?

**Оценка: уровень 3.**

Метаданные уже управляют поведением: тип карточки влияет на иконки и picker, теги `player` меняют режим добавления на карту, parent chain исключает технические дубли из некоторых списков, aliases участвуют в wiki-links.

Ограничение: это еще не централизованный metadata engine, потому что нет `PageRepository / PageIndex`.

Доказательства:

- `js/editor/blocks/blockContract.js`

Доказательства:

## Главные P0/P1 выводы

1. **PageRepository / PageIndex становится центральным архитектурным следующим шагом.**  
   Он поднимет зрелость metadata, master data, lookup, navigation, graph, map picker, wiki-links и performance.

2. **Safe HTML Boundary / Sanitizer нужен до любой интернет-версии.**  
   Сейчас безопасность HTML находится на уровне 1-2. Для local-first MVP это терпимо, для web/cloud - нет.

3. **CI нужен, чтобы локальные проверки стали обязательными.**  
   `npm run verify` и `npm run test:browser` уже существуют, осталось закрепить их в GitHub Actions.

4. **Asset Architecture стала важнее из-за новой идеи с музыкой.**  
   Плейлисты локаций из workspace нельзя делать как разовый UI. Нужны правила audio assets, asset refs, broken refs, media performance и связь с локациями.

5. **Performance карты нужно переводить из реактивной оптимизации в измеримую.**  
   Нужны performance notes, budgets и сценарии: большая карта, много токенов, фигуры, fog, presentation sync.

6. **Release process нужно формализовать.**  
   Нужны `CHANGELOG.md`, release checklist, tags и правило, как версии вроде `4.2.0` соотносятся с `package.json`.

## Обновленный приоритетный порядок развития

1. **PageRepository / PageIndex.**  
   Причина: это разблокирует масштабирование мира, поиск, wiki-links, карту, player/original tokens, шаблоны, tags, parent chain.

2. **Safe HTML Boundary / Sanitizer.**  
   Причина: это главный security blocker перед web/cloud.

3. **CI на GitHub Actions.**  
   Причина: тесты уже есть, но они должны запускаться автоматически.

4. **Asset Lifecycle Contract.**  
   Причина: картинки уже важны, а новая идея с музыкой добавляет audio assets и плейлисты.

5. **Performance Strategy для карты.**  
   Причина: карта уже тяжелая подсистема; нужно измерять, а не только чинить лаги.

6. **Release Process / Changelog.**  
   Причина: версии активно используются в коммитах, но не закреплены как release system.

7. **UX/Onboarding слой.**  
   Причина: продукт становится функционально богатым, но вход для нового пользователя пока слабый.

8. **Knowledge Graph развитие.**  
   Причина: wiki-links и backlinks есть, но graph пока не стал инструментом мышления.

9. **Desktop Adapter Plan.**  
   Причина: local-first продукт естественно просится в desktop, но нужен план adapter interfaces.

10. **AI Onboarding Guide.**  
   Причина: проект уже развивает AI-поддерживаемость; нужен отдельный краткий входной документ для новой AI/разработчика.

## Финальный вывод

По строгой модели v4 проект уже нельзя считать хаотичным прототипом. Он находится в переходной зоне между **уровнем 2** и **уровнем 3**, а по некоторым практикам достигает **уровня 4**:

- roadmap и техдолг;
- block/runtime contract;
- testing artifacts;
- local-first data ownership;
- offline core workflows.

Главная зрелость проекта сейчас - **архитектурная осознанность**. Главная незрелость - **отсутствие автоматизированных и безопасных границ**: CI, sanitizer, PageIndex, asset integrity, performance budgets.

Если двигаться по приоритетам выше, следующий качественный скачок будет не в количестве функций, а в том, что проект станет устойчивой локальной платформой для DnD-мира: с понятным ядром данных, безопасным HTML, надежным тестовым контуром, управляемыми assets и готовностью к desktop/web evolution.
