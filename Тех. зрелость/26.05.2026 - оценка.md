# 26.05.2026 - оценка технологической зрелости

Источник оценки: `Тех. зрелость/Strict_Product_Maturity_Model_v4_Evidence_Based.docx`.

Проект: **My own world**.

Дата оценки: **26.05.2026**.

## Шкала

- **Уровень 1** - отсутствует или хаотично.
- **Уровень 2** - есть частично, но риск остается высоким.
- **Уровень 3** - формализовано и реально используется.
- **Уровень 4** - системно, повторяемо и частично измеримо.
- **Уровень 5** - зрелая масштабируемая практика.
- **Неприменимо** - критерий сейчас не относится к текущему local-first режиму.

## Краткий итог

Текущая оценка проекта: **между уровнем 3 и 4**, ближе к **3.6**.

По сравнению с оценкой от 25.05.2026 проект заметно продвинулся в инженерной зрелости:

- появился обязательный CI-контур GitHub Actions;
- `PageRepository / PageIndex` стал центральным слоем lookup для ключевых сценариев;
- Safe HTML boundary получил save/load/paste sanitizer и browser security regression;
- карта переведена на data-first save через `CampaignMapModel`;
- у карты появились model-first инициативы и слои;
- крупные JS-файлы начали реально резаться на подсистемы;
- `docs/PLANS_AND_TECH_DEBT.md` стал единым backlog-планом, а исторический лог вынесен в `docs/WORK_LOG.md`;
- manual и README используются как источник входа в проект.

Главные ограничения зрелости сейчас:

- desktop/web/cloud адаптеры еще не реализованы, только спроектированы как направление;
- CD и release packaging почти отсутствуют;
- нет полноценного backup/recovery flow для workspace;
- performance governance карты пока на уровне первых сценариев и budgets, не на уровне постоянного профилирования;
- UX/onboarding пока слабее архитектуры;
- plugin/extension readiness пока только архитектурная перспектива.

## Подробная оценка

### 1.1 Cloud Native

**Вопрос 1. Насколько продукт независим от конкретного окружения запуска?**

Оценка: **уровень 3**.

Почему: ядро уже частично отделено от DOM и окружения: карта имеет `CampaignMapModel`, `CampaignMapStore`, serializer и render adapter; task tracker имеет собственную модель; PageRepository работает как отдельный слой lookup. Это доказывают `js/editor/campaignMapModel.js`, `js/editor/campaignMapStore.js`, `js/taskTracker/taskTrackerModel.js`, `js/repository/pageRepository.js`. Но storage, workspace permissions и assets все еще завязаны на браузерный File System Access API: `js/storage/persistence.js`, `js/storage/workspaceStorage.js`, `js/storage/assetStorage.js`. Значит перенос в desktop/cloud возможен без полного rewrite, но адаптеров еще нет.

**Вопрос 2. Насколько продукт готов к будущему переходу в desktop app?**

Оценка: **уровень 3**.

Почему: desktop-направление понятно и уже стоит в плане, но прототипа нет. В `docs/PLANS_AND_TECH_DEBT.md` есть блок Desktop Adapter Plan, а архитектура постепенно отделяет model/store/serializer от UI. Для уровня 4 нужен Tauri/Electron spike, `StorageAdapter`, `AssetAdapter` и desktop smoke checklist.

### 1.2 Управление архитектурой продукта

**Вопрос 1. Есть ли архитектурные правила, которые реально ограничивают хаос?**

Оценка: **уровень 4**.

Почему: правила не только записаны, но уже влияют на разработку. Есть `BLOCK_SYSTEM_CONTRACT.md`, `SAFE_HTML_CONTRACT.md`, `EDITOR_HISTORY_CONTRACT.md`, `FORMATTING_SERVICE_CONTRACT.md`, `ASSET_LIFECYCLE_CONTRACT.md`; после изменений обновляются `README.md`, `WORK_LOG.md`, manual и план. `npm run verify` проверяет syntax, unit tests, import path case, docx integrity и diff. Для уровня 5 нужны ADR, dependency graph и более строгие автоматические architecture boundary checks.

**Вопрос 2. Есть ли единый источник правды по архитектуре?**

Оценка: **уровень 4**.

Почему: `docs/PLANS_AND_TECH_DEBT.md` стал единым backlog-планом, а `docs/WORK_LOG.md` хранит историю. README описывает основные runtime/persistent правила, карту, save queue и тесты. Manual генерируется в `docs/MY_OWN_WORLD_FULL_MANUAL.docx`. Ограничение: документация еще не связана как полноценный architecture portal, а часть знаний остается в последовательности work log.

### 1.3 Интеграция, техстек, капабилити

**Вопрос 1. Насколько техстек последовательный и управляемый?**

Оценка: **уровень 3**.

Почему: проект стабильно использует vanilla JS modules, CSS, Node test runner и Playwright. Это видно в `package.json`, `playwright.config.mjs`, `tools/run_checks.mjs`. Зависимости немногочисленны и не растут хаотично. Для уровня 4 нужен dependency decision checklist и регулярный audit dependency/security.

**Вопрос 2. Можно ли добавлять новые подсистемы без разрушения существующих?**

Оценка: **уровень 4**.

Почему: task tracker, campaign map, layers, initiative, PageRepository и Safe HTML boundary добавлены как отдельные подсистемы. Разрез п.14 усилил это: `editor.js`, `toolbar.js`, `blockContract.js`, `campaignMapPresentation.js`, `tables.js` стали фасадами/контроллерами, а логика ушла в отдельные файлы. Для уровня 5 нужен внутренний module registry/plugin lifecycle.

### 2. Управление данными

**2.1 Метаданные.**

Оценка: **уровень 3**.

Почему: у страниц есть `id`, `title`, `type`, `tags`, `aliases`, `parentId`; карта хранит `template`, `grid`, `fog`, tokens, shapes, layers, initiative. Метаданные управляют поведением: picker карты фильтрует по type/tags, wiki-links ищут по title/aliases, сущности карты различают source modes. Доказательства: `js/repository/pageIndex.js`, `js/editor/campaignMapDataSerializer.js`, `js/editor/campaignMapPicker.js`.

**2.2 Качество данных и silent corruption.**

Оценка: **уровень 4**.

Почему: есть `sanitizePersistentHTMLOnSave()`, `sanitizePersistentHTMLOnLoad()`, plain text paste sanitizer, serializer блоков, data-first serializer карты, write queue. Security regression покрывают forbidden tags, runtime leakage и malformed HTML: `tests/browser/safe-html.spec.mjs`. Карта проверяется save/reload browser smoke: `tests/browser/campaign-map-data.spec.mjs`. До уровня 5 не хватает schema validation/recovery policy для всех persistent форматов.

**2.3 Master / Reference Data.**

Оценка: **уровень 3.5**.

Почему: `PageRepository / PageIndex` стал master lookup слоем для id/title/aliases/parent/type/tags, но не все старые прямые обращения к `state.pages` полностью ликвидированы. Доказательства: `js/repository/pageRepository.js`, `js/repository/pageIndex.js`, `tests/pageRepository.test.mjs`, `tests/pageIndex.test.mjs`, план 6 в `docs/PLANS_AND_TECH_DEBT.md`.

**2.4 Data Governance.**

Оценка: **уровень 3**.

Почему: lifecycle создания/переименования/переноса/удаления страниц уже отражается через repository updates и tests. Но нет полноценного audit trail изменений, restore/versioning и formal data governance process.

**2.5 Аудит данных.**

Оценка: **уровень 2**.

Почему: изменения можно отследить через git и work log, но внутри приложения нет журнала операций пользователя: кто/когда изменил страницу, удалил токен или переименовал сущность. Это станет важным перед multi-user/cloud.

**2.6 Data Contracts.**

Оценка: **уровень 4**.

Почему: у ключевых систем появились контракты и тесты: `SAFE_HTML_CONTRACT.md`, `BLOCK_SYSTEM_CONTRACT.md`, `ASSET_LIFECYCLE_CONTRACT.md`, `campaignMapDataSerializer.test.mjs`, `taskTrackerModel.test.mjs`. Ограничение: нет единого schema validator для всех страниц.

### 3-4. Security

**Общая модель безопасности.**

Оценка: **уровень 3**.

Почему: для local-first режима проект уже защищает HTML-границу через sanitizer и browser regression. Но web/cloud threat model, auth, encryption, permissions model, CSP и dependency audit пока не реализованы.

**Безопасность workspace и пользовательских данных.**

Оценка: **уровень 3**.

Почему: данные принадлежат пользователю и лежат в workspace; запись идет через `writeQueue.js`; app не требует сети. Но нет backup/recovery wizard, encryption и integrity scan всего workspace.

**Security testing.**

Оценка: **уровень 4**.

Почему: `tests/browser/safe-html.spec.mjs` ловит forbidden tags, unsafe attributes, dangerous URLs, runtime leakage и malformed HTML. Для уровня 5 нужен security-focused CI policy, dependency scanning и threat model.

**Secrets lifecycle.**

Оценка: **неприменимо / уровень 3 для текущего local-first режима**.

Почему: продукт сейчас не хранит cloud secrets/API keys. Правильное состояние - не вводить secrets до появления cloud/AI integrations. При появлении сети нужен отдельный secrets contract.

## Главные P0/P1 выводы

1. **Tables Contract** - следующий естественный шаг после разреза `tables.js`: зафиксировать правила таблиц и покрыть resize/selection/paste/keyboard regression.
2. **CSS Separation** - архитектура JS заметно окрепла, теперь CSS становится следующей зоной хаоса.
3. **Release Process / Rollback** - CI уже есть, но выпуск и откат изменений еще не являются процессом.
4. **Desktop Adapter Plan** - проект созрел до desktop spike, но нужно не "завернуть как есть", а ввести `StorageAdapter` и `AssetAdapter`.
5. **Asset UI** - broken/orphan assets уже можно вычислять, но пользователю пока негде это увидеть.
6. **UX / Onboarding** - продукт становится мощным, но новый пользователь пока слишком зависит от объяснений вне приложения.
7. **Performance Governance карты** - первые budgets есть, но нужна регулярная измеримость на реальных больших картах.
8. **Knowledge Graph / Relationships** - доменная модель мира просит явных typed relationships и graph-представление.

## Вывод

Проект уже перешел из состояния "сложное локальное приложение" в состояние **управляемой инженерной системы**. Самый сильный прогресс последних изменений - не отдельные фичи, а появление устойчивых границ: repository, model-first карта, sanitizer, contracts, CI, browser regression и разрез крупных файлов.

Главный риск теперь другой: не "все сломается от следующей функции", а "систем станет слишком много, и без контрактов/тестов/UX-слоя ими будет тяжело пользоваться". Поэтому ближайшие работы должны идти не только в сторону новых возможностей, но и в сторону формализации: таблицы, CSS ownership, release process, desktop adapters и onboarding.
