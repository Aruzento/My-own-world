# Оценка технической зрелости проекта

Дата: 01.06.2026

Основа оценки: `Тех. зрелость/Strict_Product_Maturity_Model_v4_Evidence_Based.docx`, текущий аудит файлов, актуальный план работ, результаты стресс-проверок и состояние репозитория после последних изменений карты.

## Краткий результат

Итоговая оценка: **3.8 / 5**.

Проект уже вышел из стадии хрупкого локального прототипа: есть модульная архитектура, GitHub Actions, unit/browser regression, Safe HTML boundary, PageRepository/PageIndex, data-first карта кампании, model-first таск-трекер, документация, аудит файлов и ручной/автоматический smoke-процесс. При этом до зрелости уровня 4.5-5.0 еще не хватает формального schema/versioning слоя, backup/restore сценариев, полноценной performance observability, завершенного разделения крупных файлов, desktop/cloud adapter и более жесткой политики данных workspace.

## Проверки и стресс-тесты

- `npm run test` прогнан 3 раза подряд: **55 unit-тестов прошли каждый раз**.
- `npm run test:browser` прогнан 3 раза подряд: **22 browser-теста прошли каждый раз**.
- `npm run verify`: **прошел полностью**. Проверены JS-синтаксис, импортные пути, unit-тесты, `git diff --check` и целостность `docs/MY_OWN_WORLD_FULL_MANUAL.docx`.
- Модельный стресс карты: создано 250 токенов, 120 фигур, 120 операций тумана и 20 locked fog zones. Сериализация модели заняла около **0.34 ms**, но performance budgets ожидаемо показали превышение по `visibleTokenCount` и `visibleShapeCount`.
- Проверка маркеров сломанной кодировки по исходникам и документации: критичных совпадений не найдено. Отдельно исправлен генератор аудита файлов, чтобы новые аудиты создавались в нормальном UTF-8.

## Оценка по областям

### 1. Архитектура и модульность: 4.0 / 5

Сильные стороны:

- Карта кампании уже разделена на модель, store, serializer, render adapter, pointer controller, tokens, shapes, fog, layers, initiative, presentation sync и toolbar. Доказательства: `js/editor/campaignMapModel.js`, `js/editor/campaignMapStore.js`, `js/editor/campaignMapDataSerializer.js`, `js/editor/campaignMapRenderAdapter.js`, `js/editor/campaignMapPointerController.js`.
- Таск-трекер вынесен в самостоятельную подсистему. Доказательства: `js/taskTracker/taskTrackerModel.js`, `js/taskTracker/taskTrackerRender.js`, `js/taskTracker/taskTrackerDnd.js`.
- PageRepository/PageIndex уже закрывает хаотичный lookup по `state.pages` в ключевых местах. Доказательства: `js/repository/pageRepository.js`, `js/repository/pageIndex.js`, `docs/PAGE_REPOSITORY_CONTRACT.md`.

Ограничения:

- Часть файлов все еще крупная или концептуально перегруженная: `js/editor/toolbar.js`, `js/editor/editor.js`, `js/editor/campaignMapPresentation.js`, `styles/campaign-map.css`, `styles/popup.css`.
- Некоторые подсистемы уже разделены, но еще не имеют одинаково строгого controller/model/repository контракта.

### 2. Управление данными и сохранение: 3.7 / 5

Сильные стороны:

- Карта переведена на data-first save, то есть основой сохранения является модель, а не случайное состояние DOM. Доказательства: `js/editor/campaignMapModel.js`, `js/editor/campaignMapDataSerializer.js`, `tests/campaignMapDataSerializer.test.mjs`, `tests/browser/campaign-map-data.spec.mjs`.
- Runtime UI отделяется от persistent HTML через contract/sanitizer. Доказательства: `docs/BLOCK_SYSTEM_CONTRACT.md`, `js/editor/safeHtmlSanitizer.js`, `docs/SAFE_HTML_CONTRACT.md`.
- Есть очередь записи и storage-модули. Доказательства: `js/storage/writeQueue.js`, `js/storage/pageStorage.js`, `js/storage/workspaceStorage.js`.

Ограничения:

- Нет полноценного schema validation на уровне workspace/page/map/task tracker.
- Нет встроенного backup/restore, snapshot history или recovery flow при повреждении файлов.
- Asset lifecycle описан и частично покрыт тестами, но еще не является обязательным gate для всех операций удаления/замены assets.

### 3. Тестирование и CI: 4.2 / 5

Сильные стороны:

- Есть GitHub Actions verify pipeline. Доказательство: `.github/workflows/verify.yml`.
- Есть unit-тесты моделей и helper-слоев. Доказательства: `tests/*.test.mjs`.
- Есть browser regression для карты, презентации, дерева, safe HTML, форматирования, таблиц, шаблонов и таск-трекера. Доказательства: `tests/browser/*.spec.mjs`.
- Есть запуск через `tools/run_checks.mjs` и `tools/run_browser_smoke.mjs`.

Ограничения:

- Browser-тесты покрывают основные сценарии, но не все визуальные регрессии и не весь ручной UX.
- Нет отдельного stress/performance CI-gate на тяжелые карты.
- Нет screenshot-baseline или visual regression.

### 4. Безопасность и Safe HTML: 3.7 / 5

Сильные стороны:

- Есть Safe HTML contract и sanitizer на критичные сценарии. Доказательства: `docs/SAFE_HTML_CONTRACT.md`, `js/editor/safeHtmlSanitizer.js`, `tests/browser/safe-html.spec.mjs`.
- Runtime controls не должны попадать в persistent HTML.

Ограничения:

- Для интернет-версии нужен более строгий threat model: XSS, импорт чужих workspace, unsafe URL, content security policy, permissions и auth.
- Сейчас проект local-first, поэтому часть рисков браузерной публикации еще не закрыта.

### 5. Производительность карты: 3.2 / 5

Сильные стороны:

- Есть performance strategy и performance helper. Доказательства: `docs/CAMPAIGN_MAP_PERFORMANCE_STRATEGY.md`, `js/editor/campaignMapPerformance.js`, `tests/campaignMapPerformance.test.mjs`.
- Большая часть состояния карты теперь находится в модели, что упрощает будущие оптимизации.

Ограничения:

- Стресс-проверка с 250 токенами и 120 фигурами показывает превышение текущих budget по видимым объектам.
- Туман, презентация и массовая отрисовка все еще требуют дальнейшей оптимизации: dirty rectangles, throttling, viewport culling, batch DOM updates.

### 6. UX и продуктовая целостность: 3.6 / 5

Сильные стороны:

- Есть понятные сущности: карточки, карта кампании, таск-трекер.
- Есть onboarding checklist и sample workspace. Доказательства: `docs/UX_ONBOARDING_CHECKLIST.md`, `docs/sample-workspace/`.
- Стиль постепенно унифицируется через общие UI-паттерны.

Ограничения:

- Не хватает встроенного tutorial прямо в приложении.
- Не все сложные функции объяснены пользователю внутри интерфейса: карта, слои, инициатива, туман, шаблоны, свойства.
- Визуальные регрессии пока проверяются в основном руками.

### 7. Документация и сопровождаемость: 4.0 / 5

Сильные стороны:

- Есть большой manual, README, контракты, планы, work log, аудит файлов и maturity assessment. Доказательства: `docs/MY_OWN_WORLD_FULL_MANUAL.docx`, `README.md`, `docs/PLANS_AND_TECH_DEBT.md`, `docs/PROJECT_FILE_AUDIT.md`.
- Есть AI onboarding для будущей работы Codex. Доказательство: `docs/AI_ONBOARDING.md`.

Ограничения:

- Документация требует дисциплины обновления после каждой крупной подсистемы.
- Старые исторические документы могут содержать устаревшие решения, поэтому главный источник правды должен оставаться в актуальном плане.

### 8. Release process и зрелость поставки: 3.5 / 5

Сильные стороны:

- Есть release process, changelog и CI. Доказательства: `docs/RELEASE_PROCESS.md`, `CHANGELOG.md`, `.github/workflows/verify.yml`.

Ограничения:

- Нет автоматического release/tag flow.
- Нет rollback-проверки на реальном workspace.
- Версии иногда исторически расходились между сообщениями коммитов и продуктовой логикой, это нужно ужесточить.

### 9. Готовность к desktop/internet: 3.0 / 5

Сильные стороны:

- Local-first подход хорошо подходит для будущего desktop adapter.
- Есть desktop adapter plan. Доказательство: `docs/DESKTOP_ADAPTER_PLAN.md`.

Ограничения:

- Для desktop нужны StorageAdapter/AssetAdapter spike, packaged app, file permissions, backup и migrations.
- Для internet нужны backend/API, auth, sync model, multi-user conflict resolution, secure asset hosting и server-side sanitization.

## Вывод

Текущий проект находится в уверенной зоне **3.8 / 5**: это уже не хаотичный прототип, а активно развивающийся local-first продукт с тестами, архитектурными контрактами и несколькими самостоятельными подсистемами. Главный риск сейчас не в отсутствии функций, а в росте сложности: карта, editor, таблицы, assets и workspace storage должны получить более строгие контракты, валидацию и performance gates.

Ближайший путь к уровню **4.2 / 5**: ввести schema validation/recovery для workspace, усилить performance tests карты, завершить разрез крупных файлов и закрепить visual/browser regression для сложного UI.

Путь к уровню **4.5+ / 5**: desktop adapter, backup/restore, миграции данных, visual regression, полноценная asset lifecycle automation и более строгий security model для будущей интернет-версии.
