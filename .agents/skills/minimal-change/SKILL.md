# Minimal Change Skill

Адаптация подхода Ponytail под MyOwnWorld. Идея не в том, чтобы писать меньше любой ценой, а в том, чтобы не добавлять лишнюю архитектуру, лишние зависимости и лишние пользовательские поверхности там, где проект уже умеет решить задачу проще.

Источник идеи: `https://github.com/DietrichGebert/ponytail`.

## Когда Использовать

Использовать перед задачами, где есть риск разрастания решения:

- новая подсистема, новый блок, новый сервис или новый workflow;
- рефакторинг крупного файла;
- UI-фича, которую можно решить существующим popup, model API или блоком;
- исправление бага, где соблазн переписать весь слой;
- code review на переусложнение;
- задача с формулировкой "сделай красиво/правильно/полностью", где сначала нужно найти самый маленький безопасный шаг.

Не использовать как повод игнорировать явное требование владельца продукта. Если пользователь просит весь пункт или все подпункты, сначала выполнить объем задачи, но выбрать минимально достаточную реализацию внутри этого объема.

## Лестница Решения

Перед кодом пройти сверху вниз и остановиться на первом рабочем уровне:

1. **Не строить**: задача действительно нужна сейчас, или достаточно документации, настройки, теста, удаления старого пути?
2. **Использовать существующий проектный слой**: есть ли уже `PageRepository`, `PropertiesModel`, `CharacterModel`, `CampaignMapModel`, `StorageAdapter`, popup manager, block contract или другой локальный API?
3. **Использовать платформу**: HTML/CSS/браузер/Tauri/Node уже дают нужное поведение без новой библиотеки?
4. **Использовать уже установленную зависимость**: если зависимость уже есть, можно ли применить ее без нового пакета?
5. **Сделать маленький патч**: можно ли решить точечно в одном-двух файлах без новой абстракции?
6. **Только потом строить новую модель/сервис**: если без этого нельзя сохранить контракт, тестируемость или будущую безопасность.

## Что Прочитать Перед Задачей

- `AGENTS.md`
- `docs/00-product/PRODUCT_DASHBOARD.md`
- `docs/01-delivery/PROJECT_PLAN.md`
- `docs/01-delivery/WORK_LOG.md`
- релевантный contract из `docs/02-architecture/`
- для UI: `docs/02-architecture/ui/DESIGN_SYSTEM_CONTRACT.md`
- для release-facing изменений: `release/latest/release-notes.md` и `release/latest/tester-instructions.md`

## Правила Для MyOwnWorld

- Предпочитать расширение существующей модели новому parallel-состоянию.
- Предпочитать model-first изменение DOM-патчу, если данные должны пережить save/reload.
- Не добавлять новый тип сущности, если текущие `card`, `campaignMap`, `taskTracker`, `ruleTree` или блок `Свойства` покрывают задачу.
- Не добавлять новую зависимость без явного выигрыша и без проверки, что стандартная платформа не решает задачу.
- Не создавать новый popup/controller, если можно расширить существующий popup manager или block runtime.
- Не вводить новую терминологию для пользователя, если можно углубить существующий путь.
- Удаление старого пути допустимо только после проверки совместимости и миграционного/fallback сценария.

## Что Нельзя Упрощать

- trust-boundary validation;
- backup/restore, schema recovery и data-loss handling;
- sanitizer / safe HTML boundary;
- desktop/browser storage adapters;
- accessibility и keyboard/focus для пользовательских controls;
- P0/P1 regression tests;
- release handoff, если меняется видимый сценарий;
- требования владельца продукта, явно заданные в текущей задаче.

## Как Помечать Осознанные Упрощения

Если выбран маленький временный путь с понятным потолком, оставить короткий комментарий:

```js
// minimal-change: linear scan is fine for current page count; upgrade to PageIndex lookup if this enters hot path.
```

Комментарий нужен только там, где будущему разработчику важно знать предел решения. Не засорять очевидный код.

## Что Обновить После Задачи

- `docs/01-delivery/WORK_LOG.md`, если принято архитектурное или продуктово заметное решение.
- `docs/01-delivery/PROJECT_PLAN.md`, если найден хвост или отложенный upgrade path.
- релевантный contract, если изменился путь подсистемы.
- release notes / tester instructions, если изменилось пользовательское поведение.
- manual, если изменился пользовательский сценарий.

## Проверки

- Минимум: `node tools/validate_agent_skills.mjs`, если менялся skill layer.
- Для кодовых изменений: `npm run verify`.
- Для UI/runtime: `npm run test:browser`.
- Для документации: `node tools/docs_index.mjs`.
- Для маленькой нетривиальной логики: один ближайший unit/browser regression, который упадет при поломке этой логики.

## Типовые Ошибки

- Делать новую модель вместо использования существующего контракта.
- Писать универсальный framework ради одного сценария.
- Добавлять dependency ради функции, которую дает браузер, Node или Tauri.
- Исправлять баг переписыванием всей подсистемы без необходимости.
- Называть временный shortcut полноценной архитектурой.
- Не записывать upgrade path для временного решения с известным потолком.
- Экономить на тестах там, где меняется сохранение, карта, свойства, schema, backup или desktop.
